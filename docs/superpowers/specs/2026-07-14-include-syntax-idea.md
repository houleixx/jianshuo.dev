# 字符串 include 语法（想法记录，未排期）

**状态：想法阶段，暂不实施。** 2026-07-14 与建硕的讨论记录，以后要做时从这里捡起。

## 一句话

给所有 prompt 类文本（文风、ui-config 指令、挖矿 prompt 覆盖等）加一个服务端解析的
include 语法（如 `{{include:片段/xxx}}`），使用时从全局片段库合成全文——把已验证的
「真源在代码、R2 覆盖零部署」模式从**整篇粒度**细化到**片段粒度**。

## 讨论的来龙去脉

1. 起点：建硕看了 ui-config 最新代码（内置缺省 ← 全局 R2 覆盖 ← 每用户稀疏覆盖，
   `ui-config.js` / `ui-config-custom.js`），提出文风（style）是否也该用「ref + 增量」
   而不是现在的版本法（`functions/lib/style-store.js` schema-3 全文快照链）。
2. 分析结论：版本法的两个命根子不能丢——
   - **undo/redo**（head 指针）：文风是蒸馏/偷师/手改反复调出来的，撤销是高频功能；
   - **可复现**：文章钉着「风格 vN」（`articles[i].style = N`），restyle 按 vN 重挖
     依赖每版自包含。蒸馏产出天然是全文，也拆不成「预设+增量」。
   版本法真实的痛点只有两个：预设是死拷贝（开户时种进链子，之后 `DEFAULT_STYLE`
   改进传导不到存量用户）；换预设丢个人增补。
3. 中间方案（混合）：版本条目存全文快照 + `ref`/`additions` 元数据，UI 可渲染
   「基于 XX 体 + 我的增补」，升级是显式按钮。
4. 建硕更进一步：**在字符串层支持 include，使用时解析**——比结构化 ref 更优雅：
   不改 schema、编辑体验零改动（还是纯文本）、一个机制通吃所有 prompt 文本。

## 设计要点（四个必须钉死的坑）

1. **可复现性（最大的坑）**：版本保存时**存两份**——raw（用户编辑的原文，含 include
   行）+ resolved（保存那一刻的解析快照）。当前 head 活解析（享受基座升级），
   历史版本重挖用快照。这条不做，include 就是定时炸弹。
2. **信任边界**：只解析白名单命名空间（官方片段库 `config/fragments/*`、预设）；
   prompt-share 兑换来的他人指令**一律不解析**（或剥掉）——否则是注入口。
3. **循环与深度**：解析器深度上限 2 层 + 环检测，超限原样保留当普通文本。
4. **解析点唯一**：收在现有单一真源函数（`resolveStyle`/`readStyleText`、ui-config
   instruction 出口），谁都不许自己再解析。语法要与客户端占位符
   `{{KEY}}/{{LINE}}/{{QUOTE}}`（客户端替换）长得明显不同，避免混淆替换时机。

## 性能结论

设计问题，不是性能问题。两条守住则全链路最坏 +30ms（一次 R2 读）：

- **片段整库一个文件**（`config/fragments.json` 装全部片段），不管几个 include、
  嵌套几层都只多一次读；反模式是循环里逐片段 fetch。
- **isolate 级 60s 记忆化**（全局配置低频变更，内存缓存 + TTL）。

挖矿/编辑 turn 被 LLM 调用（秒级）支配，+30ms 是噪音；历史版本重挖走快照零增量；
社区 feed 等路径不碰 prompt 文本零影响；token 成本与今天的全文快照持平。

## 能简化什么（当时盘点）

**现在就疼的：**

| # | 场景 | 痛点 |
|---|------|------|
| ① | 文风预设（style-store.js） | 预设升级传导、换预设保增量 |
| ② | ui-config 指令重复尾巴 | 「正文其他内容都不要动」×8、photo 句式 ×6，改一处漏一处 |
| ③ | 挖矿 prompt 用户覆盖 | 整篇 fork 漂移，官方升级错过；include = 官方基座+我的三行 |
| ④ | 跨 prompt 一致性规则 | 「称呼 AI 用他」「始终中文」散在挖矿/编辑/采访员/追问各写一遍 |
| ⑤ | eval 实验粒度（/wjs-evaling-voicedrop-prompts） | candidate = 只换一个片段，归因清晰，赢了改 R2 零部署 |

**未来的：** ⑥ 用户自定义片段库；⑦ 指令市场引用官方片段（官方升级自动流到兑换者）；
⑧ 中英 prompt 家族共享骨架（对抗中英两套 prompt 漂移）；⑨ 片段级 A/B（编号实验
方法论搬过来，片段是对的实验单位）。

**明确不用的地方：** 客户端占位符（运行时数据填充，不是文本复用）；App 内文案
（String Catalog 管）；兑换来的他人指令（禁解析）。

## 建议的 v1 范围（当时的估计）

解析器（~30 行 + 环检测 + 测试）、`config/fragments.json` 全局片段库、三个文风预设
入库、`resolveStyle` 与 ui-config instruction 出口接上、版本保存落 resolved 快照、
prompt-share 禁 include。不做用户自定义片段、不做参数化 include（v2）。

优先级建议：先 ②④（纯还技术债零风险）→ ①③（产品语义变化）→ 其余自然生长。
